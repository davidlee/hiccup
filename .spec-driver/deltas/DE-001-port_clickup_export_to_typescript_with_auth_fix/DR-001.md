# DR-001 – Port clickup-export to TypeScript with auth fix

## 1. Executive Summary

- **Delta**: [DE-001](./DE-001.md)
- **Status**: draft
- **Synopsis**: Replace Python clickup-export with TypeScript. Fix auth crash, eliminate N+1 API calls, add pagination.

## 2. Problem & Constraints

- **Current Behaviour**: Python script fetches task list, then re-fetches each task individually to get subtasks. Any 401/403 response aborts the entire export. No pagination — lists >100 tasks are silently truncated.
- **Drivers**: Org standardisation on TypeScript; tool is unusable when token has partial permissions.
- **Constraints**: Must produce a useful JSON export of task hierarchies. No other constraints — no users, no backwards compat.

## 3. Architecture Intent

- **Target Outcomes**:
  - Single `clickup-export` CLI command that reliably exports task trees
  - Graceful degradation on permission errors
  - Complete data for large lists (pagination)
- **Guiding Principles**:
  - Small modules, single responsibility
  - Pure functions for tree assembly (no I/O)
  - API client handles all HTTP concerns (auth, pagination, error boundaries)
  - CLI layer is thin orchestration

## 5. Design Detail

### API Client (`src/clickup.ts`)

- Single `fetchClickUp(path, token)` that wraps fetch with:
  - Authorization header
  - Non-2xx → return `{ ok: false, status, error }` (not throw)
  - Caller decides whether to skip or abort
- `fetchAllTasks(listId, token)` — paginated loop over `GET /api/v2/list/{id}/task`:
  - Query: `archived=false`, `include_closed=true`, `subtasks=true`, `page=N`
  - Accumulate until page returns empty
  - Per-page errors: warn and return what we have so far
- `fetchTask(taskId, token)` — single task fetch for task-specific export entry point

### Tree Assembly (`src/tree.ts`)

- Pure function: `buildTree(tasks: Task[]): TreeNode[]`
- Index tasks by ID, group children by `parent` field
- Return forest of root nodes (where `parent` is null/absent)
- Tasks whose parent ID isn't in the set become roots (handles permission gaps)

### Export (`src/export.ts`)

- `exportToDir(tree, tasks, outputDir)`
- Write `tree.json` (the nested hierarchy)
- Write `tasks/{id}.json` for each task in the flat list
- Create directories as needed

### CLI (`src/cli.ts`)

- Parse positional arg (ClickUp URL)
- Optional second arg for output dir (default: `clickup-export`)
- Read `CLICKUP_API_TOKEN` from env
- Parse URL → extract list ID and optional task ID
- If task ID: fetch that task + subtasks, build subtree
- If list ID: fetch all tasks (paginated), build tree
- Call export

### URL Parsing

- Same logic as Python: extract list ID from `/l/{id}` path segment, task ID from `pr` query param
- Straightforward regex + URL parsing — no library needed

## 7. Design Decisions & Trade-offs

- **DEC-001**: Use ClickUp API v2 for task operations. v3 only covers chat/docs/attachments — no task list/fetch endpoints exist in v3.
- **DEC-002**: Build tree from flat list. The list endpoint with `subtasks=true` returns all tasks (parents + children) in a flat array with `parent` fields. Assembling the tree client-side eliminates N+1 API calls and is a pure function.
- **DEC-003**: Per-request error boundaries. API errors (401, 403, 5xx) on individual requests log a warning to stderr and skip that resource. Only total auth failure (list fetch itself fails) aborts.
